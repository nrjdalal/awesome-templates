[Better Auth](https://better-auth.com) owns the whole auth surface (sign-in methods, sessions, organizations, and roles) as one config in `packages/auth/src/index.ts`, backed by the same Drizzle schema as the rest of your data. It runs in-process against your own Postgres, so no user records leave your database and there is no separate auth service to keep in sync.

## Sign-in methods

The sign-in dialog renders itself from whatever is configured. It fetches `GET /api/auth/providers`, which returns `enabledProviders`, and draws only those buttons, so turning a method on or off is usually an env change, not a UI change.

**OAuth (GitHub and Google) is optional.** A provider registers only when _both_ halves of its credential pair are set, so a fork can ship with GitHub, Google, both, or neither:

- **GitHub**: `GITHUB_CLIENT_ID` + `GITHUB_CLIENT_SECRET`
- **Google**: `GOOGLE_CLIENT_ID` + `GOOGLE_CLIENT_SECRET`

`packages/auth/src/index.ts` builds `socialProviders` conditionally from those vars and exports the resulting `enabledSocialProviders`. Adding a brand-new provider is a small code change: register it conditionally there, then add a button in `web/next/src/components/access.tsx`.

**Magic link is off by default.** The client plugin is registered but the server `magicLink` plugin is not, so the email field stays hidden rather than showing a dead control. To enable it, add `magicLink({ sendMagicLink })` to the `plugins` array in `packages/auth/src/index.ts` and implement `sendMagicLink` to deliver the email.

**Agent sign-in** is a local-only shortcut: `POST /api/agents/sign-in-as` mints a real session as `LocalAgent` with the `admin` role, gated to the `local` environment and a trusted `Origin`. See [Working with Agents](/docs/getting-started/working-with-agents).

<Callout type="warn" title="Configure at least one method before shipping">

A deployed fork with no OAuth providers and no magic link shows "No sign-in options are configured yet." in the sign-in dialog; the agent sign-in is hidden in production. Wire up a provider or magic link before exposing a login surface.

</Callout>

## Callback URLs

Each OAuth app needs the redirect URL that Better Auth (mounted at `/api/auth`) listens on. The API runs on port 4000 locally, so the callback path is `/api/auth/callback/<provider>` on your API origin:

- **Local**: `http://localhost:4000/api/auth/callback/github` and `http://localhost:4000/api/auth/callback/google`
- **Production**: the same `/api/auth/callback/<provider>` path on your deployed API origin, e.g. `https://api.example.com/api/auth/callback/github`

This is the value for GitHub's "Authorization callback URL" and Google's "Authorized redirect URI".

## The admin role and /console

`/console` is the privileged area, gated by a single `role` column on the `user` table (added by the Better Auth [Admin plugin](https://www.better-auth.com/docs/plugins/admin)). The guard in `web/next/src/lib/auth/console.ts` grants access only when `user.role === "admin"`; everyone else keeps the default `user` role and gets a `notFound()`. There is no env allow-list: the role _is_ the source of truth. The same plugin adds ban columns (`banned`, `banReason`, `banExpires`) to `user` and lets an admin impersonate another user, recording who did it in `session.impersonatedBy`.

Bootstrap the first admin from the CLI:

```bash
bun run console:roles grant you@example.com   # also: revoke <email>, list
```

The guard reads the session with the cookie cache disabled, so a grant or revoke takes effect on the very next request.

## Organizations and teams

The [Organization plugin](https://www.better-auth.com/docs/plugins/organization) (with `teams` enabled) makes the app multi-tenant. Users create and switch organizations from the dashboard sidebar; the last active org is stored in a cookie and restored on next login. Members carry a `role` within an org (default `member`), invitations go out by email, and teams group members inside an org. The backing tables (`organization`, `member`, `team`, `teamMember`, `invitation`) live in [the schema](/docs/manage/database).

Read the current context on the client:

```ts
import { authClient } from "@/lib/auth/client"

const { data: orgs } = authClient.useListOrganizations()
const { data: activeOrg } = authClient.useActiveOrganization()
await authClient.organization.setActive({ organizationId: "..." })
```

## Sessions and cookies

Sessions live in the `session` table and travel as a secure cookie; the active `organizationId` and `teamId` ride along on the row. Cross-subdomain cookies switch on automatically when `HONO_APP_URL` is a subdomain. Better Auth caches the session in a cookie for `maxAge: 300` (~5 minutes), so a role change made elsewhere can lag by that much unless the session is revoked.

Reading the session is one call on either side:

```ts
// server component or layout
import { auth } from "@/lib/auth"
const session = await auth.api.getSession()

// client component
import { authClient } from "@/lib/auth/client"
const { data: session } = authClient.useSession()
```

The `(protected)` layout (`web/next/src/app/(protected)/layout.tsx`) redirects unauthenticated users server-side; API routes validate the session in the auth middleware (`api/hono/src/middlewares/auth.ts`).
