# Alembic Migration Management — Detailed Procedure

## Default Flow Position

This skill participates in the [Default Coding Flow](../../../../AGENTS.md#default-coding-flow) at the **`implement`** step and contributes directly to the **`verify`** step (autogenerated revisions must be applied and re-tested).

`approach options` upstream is **conditionally mandatory**: required when the schema change touches indexed columns, alters constraints affecting transactional behaviour, or runs a backfill. Skip for additive nullable-column changes that follow established Alembic patterns.

After implementation, route to:
- `verify` — `alembic upgrade head` against a clean DB **and** `/test-domain {name} run`
- `self-review` — manual diff review of the autogenerated revision body, plus the **Zero-Downtime Safety Review** below ([ADR 056](../../../history/056-zero-downtime-migration-safety.md))
- `completion gate` — `/sync-guidelines` if shared docs reference the migration

## Pre-check
1. Verify `alembic.ini` exists
2. Verify `_env/local.env` file exists and contains DB connection info
3. Verify target domain's Model file exists: `src/{name}/infrastructure/database/models/{name}_model.py`

> **Note**: `load_models()` in `migrations/env_utils.py` **automatically discovers** all models
> under `src/*/infrastructure/database/models/`. No separate import configuration needed.

## Command Mapping

### generate (Create new migration)
```bash
alembic revision --autogenerate -m "{domain}: {description}"
```
- Message format: `"{domain}: {description}"` (e.g., `"user: add email_verified column"`)
- Always show the generated file to the user and request review
- Run the migration-safety advisory checker on the new revision (see **Zero-Downtime Safety Review**)

### upgrade (Apply migration)
```bash
alembic upgrade head    # Apply up to latest
alembic upgrade +1      # Apply one step only
```

### downgrade (Rollback migration)
```bash
alembic downgrade -1    # Rollback one step
```

### status (Check current state)
```bash
alembic current         # Currently applied revision
alembic history         # Full history
```

## Workflow

### Migration after adding a new domain
1. Confirm domain Scaffolding is complete via `/new-domain {name}`
2. Review Model file: `src/{name}/infrastructure/database/models/{name}_model.py`
3. `alembic revision --autogenerate -m "{name}: initial migration"`
4. Review generated file: latest file under `migrations/versions/`
5. Ask user to verify the upgrade/downgrade functions
6. After approval, apply with `alembic upgrade head`

### Migration after modifying an existing domain model
1. Verify Model changes (compare before/after)
2. `alembic revision --autogenerate -m "{name}: {change description}"`
3. Review the generated file
4. After user approval, `alembic upgrade head`

## Zero-Downtime Safety Review

> Background: [ADR 056](../../../history/056-zero-downtime-migration-safety.md).

In a zero-downtime deployment (rolling / blue-green) a migration runs while the
previous app version still serves traffic, so review every autogenerated
revision for lock-taking or compatibility-breaking DDL before applying it.

1. Run the advisory checker on the new revision:
   ```bash
   uv run python tools/check_migration_safety.py migrations/versions/<rev>.py
   ```
   It also runs as a non-blocking `verbose` pre-commit hook. It flags a NOT NULL
   add without a safe default, a non-`CONCURRENTLY` index, a column drop/rename,
   a type change, and blocking constraint additions — against **pre-existing**
   tables (operations on a table created in the same revision are ignored, since
   an empty table cannot block live traffic).
2. For each finding, apply the safe alternative from the per-engine table in
   [`docs/operations/rdb-migrations.md`](../../../operations/rdb-migrations.md)
   (## Zero-Downtime Migrations) — usually the expand-contract split.
3. The checker is **advisory** (ADR056-G1) and never blocks. A deliberately
   staged or low-risk change may proceed once reviewed; note the rationale in
   the revision or PR.

## Cautions
- **Zero-downtime safety** -- review lock-taking / compatibility-breaking DDL before applying; see the Zero-Downtime Safety Review above and [`docs/operations/rdb-migrations.md`](../../../operations/rdb-migrations.md)
- **autogenerate is not 100% accurate** -- always review the generated file
- **Column renames are not detected** (generated as drop+add) -- manual correction needed
- Verify the `env` value in `alembic.ini` matches the environment (local/dev/stg/prod)
- **Always back up before applying to production**
- The `migrations/versions/` directory is auto-created on first revision if it doesn't exist

## Verification after upgrade/downgrade
1. `alembic current` -- verify applied revision
2. Run domain integration tests: `pytest tests/integration/{name}/ -v` (verify DB schema matches)
3. Report results to the user
